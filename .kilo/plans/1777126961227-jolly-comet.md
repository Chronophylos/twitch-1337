# Issue 64: AI Web Search / Research Agent Plan

## Scope (Confirmed)
- Add both tools: `web_search` and `fetch_url`.
- Use self-hosted SearXNG as provider.
- Add in-memory caching.
- Integrate into existing `!ai` flow (no separate `!search` command).
- Enforce strict tool-surface isolation: main `!ai` cannot access `*_memory` tools, and memory extraction/consolidation cannot access web tools.

## What Issue #64 Is About
Issue #64 asks for current-affairs/news awareness in `!ai` by letting the model fetch fresh web information at runtime. The current bot can already do tool-calling (for memory extraction), so this is an extension of existing architecture, not a new subsystem style.

## Current Architecture Constraints
- `!ai` currently calls `chat_completion()` in `src/commands/ai.rs`.
- Memory extraction already uses `chat_completion_with_tools()` with a controlled tool list from `src/memory/tools.rs`.
- Tool execution loop pattern and round-tripping already exist in `src/memory/extraction.rs`.
- `LlmClient` trait and OpenAI-compatible/Ollama backends already support tool calling in `src/llm/mod.rs`.

## Final Design

### 1) New Web Search Module
Create `src/web_search/`:
- `mod.rs`: public exports (`SearchClient`, result structs, config structs, tool defs).
- `client.rs`: SearXNG HTTP client + result normalization.
- `tools.rs`: definitions for `web_search` and `fetch_url`.
- `executor.rs`: executes web tools and returns `ToolResultMessage` text payloads.
- `cache.rs`: small in-memory TTL cache for search/fetch responses.

### 2) Provider Choice: SearXNG (Self-Hosted)
Implement SearXNG-only for now (no multi-provider abstraction in first cut).
- Endpoint from config (e.g. `https://search.example.com/search`).
- `format=json` queries.
- Normalize to compact records: `title`, `url`, `snippet`, optional `published_at`, `source`.
- Add sane defaults and strict timeouts.

### 3) Tool Surface for Main `!ai`
Add web-only tools:
- `web_search(query, max_results?)`
- `fetch_url(url)`

`fetch_url` behavior:
- Fetch page with timeout and response-size cap.
- Extract readable text (strip scripts/styles/nav noise).
- Return truncated plain text summary chunk suitable for model context.

### 4) Strict Tool Isolation (Hard Requirement)
Implement explicit, separate tool registries and executors:
- Memory extraction/consolidation continues using only `memory::extractor_tools()` / `memory::consolidator_tools()`.
- Main `!ai` uses only `web_search::ai_tools()` (web tools).
- No shared dynamic registry that can accidentally merge tool sets.
- Dispatchers reject unknown tool names with deterministic error strings.
- Add tests proving isolation in both directions.

### 5) `!ai` Integration Strategy
Upgrade `AiCommand` execution path:
- If web search disabled: keep current `chat_completion()` path.
- If enabled: run tool-calling loop (`chat_completion_with_tools`) with only web tools.
- Loop max rounds (new config knob, default 3).
- Execute returned tool calls via web executor; append tool results; continue.
- Exit on text response and send to chat as today.
- Maintain existing cooldown and response truncation behavior.

### 6) Configuration Changes
Extend `AiConfig` with optional web section (nested under AI to avoid global sprawl):
```toml
[ai.web]
enabled = false
base_url = "https://searxng.example/search"
timeout = 15
max_results = 5
max_rounds = 3
cache_ttl_secs = 300
cache_capacity = 100
user_agent_suffix = "twitch-1337-web"
```
Validation rules:
- `base_url` non-empty, valid URL.
- `max_results` in 1..=10.
- `max_rounds` in 1..=6.
- `cache_capacity` > 0.

### 7) Caching
In-memory TTL cache keyed by normalized request:
- `web_search`: key = normalized query + effective max_results.
- `fetch_url`: key = normalized URL.
- Defaults: TTL 300s, capacity 100 entries.
- Evict oldest/expired entries first.
- Cache only successful responses.

### 8) Error Handling & Safety
- Timeouts per outbound HTTP call.
- Return structured tool-result errors to the model (not silent failure), e.g. `{"error":"search_timeout"}`.
- Truncate oversized snippets/content before returning to model.
- Prevent SSRF-ish misuse in `fetch_url`:
  - Allow only `http/https`.
  - Block localhost/private/link-local ranges.
  - Optional domain allow/block list can be added later.

## Implementation Plan
1. Add config structs/defaults/validation for `[ai.web]` in `src/config.rs`.
2. Implement `src/web_search` module (client, tools, executor, cache).
3. Wire web client creation in handler setup and pass into `AiCommand`.
4. Extend `AiCommand` with web tool-call loop path.
5. Add hard isolation guards between web and memory tool surfaces.
6. Add unit tests for SearXNG parsing, fetch extraction, cache behavior, and unknown-tool rejection.
7. Add integration tests for:
   - `!ai` using web tools successfully.
   - web disabled fallback path.
   - isolation: `!ai` cannot invoke memory tools; extractor cannot invoke web tools.
8. Update `config.toml.example` with `[ai.web]` section and concise usage notes.

## Test Plan

- Integration (with mocked HTTP + fake LLM):
  - model requests `web_search` then answers with citations.
  - model requests `fetch_url` and summarizes content.
  - isolation regression tests (both directions).
- Full gate: `cargo fmt --all`, `cargo clippy --all-targets -- -D warnings`, `cargo test`.

## Success Criteria
- `!ai` can answer current-events prompts using live web data from SearXNG.
- Both `web_search` and `fetch_url` are available to main `!ai` when enabled.
- Tool-surface isolation is enforced and covered by tests.
- Caching reduces repeated outbound requests.
- Existing memory extraction behavior remains unchanged.
